# 鸿雁终端6月15日v0.9软件验证测试报告

## 执行摘要

| 项目 | 内容 |
|------|------|
| 测试日期 | 2026年6月15日 9:50起 |
| 软件版本 | **HongyanSar_v0.9**（Windows）+ **HongyanSar_v0.9.apk**（Android） |
| 测试设备 | 节点1(e548)连手机、节点2(bf9a)连电脑、节点3(6e3a)连手机 |
| 测试环境 | 室内，相互间1米左右 |
| LoRa配置 | BW125 / SF9 / CR5 |
| 中继状态 | **节点1、节点2 Relay on**（文档明确标注） |
| 测试重点 | 接收延迟20秒问题根因定位（每项测试反复多次） |
| 总测试轮次 | **26轮**（测试项一8+12轮，测试项二6轮，测试项三6轮） |

### 核心结论

1. **🔴 发现新P0问题——Windows端消息静默堆积**：手机→电脑方向，Windows端HongyanSar有3/8轮（37.5%）完全收不到消息，直到断开重连或重启软件后才一次性全部显示
2. **✅ 电脑→手机方向高度可靠**：12轮测试中8轮100%成功秒收，2~3分钟短间隔内零失败
3. **⚠️ 手机→手机发送失败率20~40%**：远高于电脑→手机的0~5%
4. **⚠️ 泛洪路由幽灵消息与时间间隔强相关**：短间隔（2~5分钟）零幽灵，长间隔（11分钟+）必现幽灵
5. **⚠️ 手机端接收延迟仍不稳定**：有时秒收、有时10秒、有时20秒，与v0.8一致未改善

### 🔴 关键发现：Windows端消息静默堆积

3/8轮（第2、6、8次）出现**Windows端完全收不到消息**的现象：

| 轮次 | 间隔时间 | 恢复方式 | 堆积消息数量 |
|------|---------|---------|-------------|
| 第2次 | 24分钟 | 手动断连→重连 | 10条 |
| 第6次 | 16分钟 | 关闭→重启软件→自动连接 | 20条 |
| 第8次 | 20分钟 | 关闭→重启软件 | 20条 |

**规律**：
- **与发送间隔时间强相关**：3次静默堆积均发生在与前次发送间隔>11分钟时
- **消息已到达硬件但App层未展示**：断连重连/重启后消息立即秒收，说明消息已在BLE缓冲区或本地缓存中，只是HongyanSar未读取/展示
- **时间戳异常**：重启后收到的消息显示"8分钟前""3分钟前"而非"刚刚"，确认消息实际到达时间远早于展示时间
- **这是v0.9新问题**：v0.6/v0.8未出现过"完全收不到"的情况，v0.8仅表现为~10秒/条延迟

## 根因分析与规律总结

### 规律1：发送失败率与方向强相关（最稳定规律）

| 发送方向 | 发送端 | 平均失败率 | 特征 |
|---------|--------|-----------|------|
| 电脑→手机 | Windows | **~4%** | 几乎全部成功，间隔2分钟以上更稳定 |
| 手机→电脑 | Android | **~16%** | 频繁失败+发送中，偶发100%失败 |
| 手机→手机（3→1） | Android | **~35%** | 最高失败率，乱序严重 |
| 手机→手机（1→3） | Android | **~10%** | 明显优于3→1 |

**推论**：失败率差异来自两个因素——(1) 发送端APP差异（Windows比Android更可靠）；(2) 节点硬件差异（节点1比节点3更稳定）。

### 规律2：Windows端消息静默堆积与空闲时间相关

```
空闲时间 < 11分钟 → 正常接收（延迟3秒~20秒）
空闲时间 ≥ 11分钟 → 37.5%概率完全收不到，需断连/重启才恢复
```

**推论**：HongyanSar Windows端的BLE轮询机制在长时间空闲后进入低功耗/休眠状态，停止主动读取BLE特征值。断连重连或重启软件重新激活BLE轮询，缓冲区消息一次性读出。

### 规律3：幽灵消息出现条件

| 条件 | 是否出现幽灵消息 |
|------|---------------|
| 连续发送无间隔 | 可能（缓存累积到阈值时） |
| 间隔2~6分钟 | 通常不出现 |
| 间隔11~22分钟 | **必然出现** |
| 间隔1小时+ | 出现，且幽灵消息编号更分散 |

**推论**：泛洪路由在中继节点（节点2）维护一个消息转发缓存，缓存条目有TTL（生存时间）。TTL到期后，缓存条目被错误地重新投递而非丢弃。间隔越长，到期条目越多，幽灵消息越多。

### 规律4：接收延迟与发送端无关，与接收端强相关

结合v0.8跨APP测试结论，v0.9进一步确认：
- 同一个接收端（如节点2 Windows），无论发送端是手机还是电脑，延迟特征一致
- 同一个发送端（如节点3手机），向不同接收端发送，延迟特征随接收端变化

### 规律5：路由标识"1跳"比"泛洪"更易失败

截图证据显示多条"1跳"消息失败而同轮"泛洪"消息成功。**"1跳"路由可能是不稳定的短暂路由状态，消息选择1跳路径时实际路径已不可达**。

## 修复建议

### 优先级1：Windows端BLE保活机制（P0 #1）

这是当前最严重的问题——消息堆积到断连/重启才显示，等于通信中断。

1. **实现BLE连接心跳**：每30秒发送一次ping/keepalive，维持GATT通知活跃
2. **实现BLE通知重订阅**：检测到长时间无数据时，主动重新订阅通知特征值
3. **实现消息轮询兜底**：即使通知未触发，每5秒主动读取一次接收特征值作为兜底

### 优先级2：发送ACK机制优化（P0 #2 + P1 #6）

1. **异步化发送确认**：当前发送端同步等待ACK，超时即标失败。改为发送后立即返回，后台等待ACK
2. **增加重传次数**：当前泛洪重传不足导致高失败率，建议增加1~2次重传
3. **1跳路由健康检查**：发送前先ping目标节点确认路径可达，避免选到已失效的1跳路径

### 优先级3：泛洪路由缓存管理（P1 #3 #4）

1. **消息ID去重**：每条消息携带唯一ID，中继节点按ID去重，避免同一消息被反复转发
2. **缓存TTL正确处理**：到期缓存必须丢弃而非重新投递
3. **跳数限制**：设置最大泛洪跳数（建议3跳），减少网络负载
4. **消息角标智能过滤**：只统计未读唯一消息数，重复消息不增加角标

### 优先级4：接收排序优化（P2 #8）

1. **消息序列号排序**：接收端按发送序列号重排后再展示
2. **排序窗口设置**：窗口时长2~3秒即可（空中传输时间远小于此）
