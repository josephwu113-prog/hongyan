# 鸿雁终端6月17日v0.12软件验证测试报告

## 执行摘要

| 项目 | 内容 |
|------|------|
| 测试日期 | 2026年6月17日 10:30起 |
| 软件版本 | **HongyanSar_v0.12**（Windows）+ **HongyanSar_v0.12.apk**（Android） |
| 测试设备 | 节点1(e548)连手机、节点2(bf9a)连电脑、节点3(6e3a)连手机 |
| 测试环境 | 室内，相互间1米左右 |
| LoRa配置 | BW125 / SF9 / CR5 |
| 中继状态 | **节点1、节点2 Relay on** |
| 测试重点 | 针对v1.0发现的问题（BLE休眠、1跳路由风暴、消息重复）验证 |
| 总测试轮次 | **12轮**（测试项一4+2轮，测试项二3轮，测试项三3轮） |

### 核心结论

1. **🔴 整体性能严重回退**：v1.0验证通过的多项指标在本版恶化，"越改越乱"
2. **🔴 发送端状态显示完全不可信**：大量标记为"发送失败"或"正在发送"的消息实际已被接收方成功收到——这是ACK机制的系统性缺陷
3. **🔴 幽灵消息（重复）爆发**：测试项一1.1第4轮10条消息收到19条（9条幽灵），为v0.6以来最严重
4. **🔴 电脑端接收延迟回退至~20秒/条**：v1.0已改善到秒级，v0.12回退到v0.9水平
5. **🔴 1跳路由风暴恶化**：电脑→手机方向第2轮13/30条消息卡"正在发送"，10条被切换为1跳路由
6. **⚠️ 手机→手机方向数据面基本正常**：测试项二、三消息实际送达率100%，但发送端状态误报严重
7. **⚠️ 新增5秒延时输入功能**（电脑端）：意图限流但效果存疑，延迟仍~20秒

## 🔴 根因分析：半个多月反复修改未果的真正原因

### 核心判断：问题不在LoRa传输层，而在BLE通信+ACK协议层

从v0.6到v0.12的6个版本测试数据揭示了一个关键事实：**LoRa空口传输是可靠的，消息确实到达了对方节点。问题出在消息到达节点硬件后，如何通过BLE传递给手机/电脑APP，以及APP如何确认消息状态。**

#### 证据链

| 证据                 | 来源                        | 结论                                   |
| ------------------ | ------------------------- | ------------------------------------ |
| 发送端报"失败"但接收端100%收到 | v0.12测试项三第1/2轮            | ACK未回传→发送端误判失败→触发重传→幽灵消息             |
| 发送端报"发送中"但接收端已收到   | v0.12测试项一1.2第1轮 #22,26,28 | 异步确认无ACK超时兜底，状态永远卡"发送中"              |
| 重连后幽灵消息爆发          | v0.12测试项一1.1第4轮 9条幽灵      | 重连时BLE缓冲区积压消息+重传消息同时涌入               |
| 电脑端~20秒延迟          | v0.12测试项一1.1全部4轮          | Windows端BLE读缓冲区频率过低，或通知机制失效          |
| 1跳路由风暴             | v1.0起出现，v0.12恶化           | 路由选择逻辑切换时，消息同时走泛洪+1跳两路，叠加ACK缺失导致重传风暴 |

### 三大根因及其关联

```
根因1: BLE连接管理缺陷
  ├── Windows端BLE通知未正确唤醒APP → 消息堆积在缓冲区 → 20秒延迟
  ├── 手机端BLE休眠(虽降低但未根除) → 偶发完全收不到
  └── 重连时缓冲区+重传消息同时释放 → 幽灵爆发

根因2: ACK/确认协议缺失或不完整
  ├── 发送端无法确认消息是否送达 → 误报"失败"/永久"发送中"
  ├── 误判失败 → 触发重传 → 同一消息多次到达 → 幽灵消息
  ├── v1.0加异步确认是方向正确，但无ACK超时兜底 → 状态卡死
  └── 接收端收到重复消息无去重 → 幽灵消息直接展示

根因3: 路由逻辑与流量控制冲突
  ├── 泛洪+1跳路由切换 → 消息走两路 → 双倍流量
  ├── 双倍流量+BLE带宽有限 → 队列溢出 → 卡"发送中"
  ├── 5秒限速/延时输入只是症状缓解 → 未解决路由切换风暴
  └── 中继节点转发重复消息 → 进一步放大幽灵消息
```

### 为什么"越改越乱"

| 版本    | 改了什么         | 解决了什么              | 引入/暴露了什么                | 根因是否触及     |
| ----- | ------------ | ------------------ | ----------------------- | ---------- |
| v0.6  | 基线           | —                  | —                       | —          |
| v0.8  | 优化接收处理       | Windows延迟从20秒降到10秒 | 手机端延迟未改善                | ❌ 只优化了症状   |
| v0.9  | 继续优化         | —                  | **BLE休眠37.5%**、手机→手机65% | ❌ 新症状出现    |
| v1.0  | 5秒限速+异步确认    | 整体成功率84%→97%       | **1跳路由风暴**、ACK不完整       | ⚠️ 限速有效但治标 |
| v0.12 | 继续调整（具体变更未知） | 手机→手机100%送达        | **延迟回退20秒、幽灵爆发、状态不可信**  | ❌ 改一处坏一处   |

**根本原因**：每次修改都是在BLE+ACK这个有结构性缺陷的基础上做局部调整。就像在流沙上盖楼——修了东墙塌西墙。BLE连接管理和ACK协议是基础设施，必须**先修好地基，再改上层逻辑**。

| ---- | -------------------------- | ----------- | ---- | ----------------------------------------------------------------------------------------------------------------------------------- |
| P0-1 | **ACK确认协议不完整**             | 全方向、全版本     | ACK层 | 1) 定义明确ACK/NACK协议：消息到达→接收端回ACK→发送端确认成功；2) ACK超时设定（建议3秒，SF9空口往返约1秒）；3) 超时未收到ACK才判定失败，允许有限重传（1次）；4) 接收端去重：按消息ID/序号去重，相同消息只展示一次        |
